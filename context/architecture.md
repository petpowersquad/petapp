## System Boundaries

- `app` — Routing & Page Shell Layer: Controls the page structural layout states.
- `trigger` — Long-running background jobs: AI design generation and spec generation.
- `components` — Reusable UI Layer: Houses raw styled presentation layers.
- `hooks` — State & Sync Connectors: Controls offline state queue handling, camera upload mutations, and local component layout interactions.
- `supabase` — Database schema and generated client output.
- `lib` — Client Engines & Utilities: Owns core application business logic, Prisma data fetch configurations, validation schemas, and external API mappings.

## Storage Model

- **Supabase PostgreSQL Database**: Holds all highly structured, relational business information. This includes user profile link tokens, multi-pet operational records (name, species, breed, age metrics, etc.), specific `scheduled_tasks` state records, and the structured data parameters generated by past triage scans.
- **Supabase Storage**: Houses the persistent physical asset images uploaded during user onboarding scans and health triage checks. Files are uniquely named via cryptographically secure UUIDv4 hashes mapping back to relational lookup keys in the main database.

## Auth and Access Model

- JWT Token Synchronization: The client authenticates directly with Clerk. Clerk issues a secure, short-lived JWT token containing the user's metadata, which is passed inside the Supabase database headers.
- Row-Level Security (RLS) Boundaries: Access control is enforced directly at the database layer using Supabase RLS policies. The database will completely reject any SELECT, UPDATE, or DELETE query on user-owned tables unless the requesting row's user_id strictly matches the verified identity extracted from the Clerk JWT.
- Isolated Relational Ownership: Every pet entry created points cleanly to a single primary user_id. Metrics, history tracking timelines, and scheduled medication schedules cannot exist detached from a valid parent user scope.
- Every project has a single owner (Clerk user ID).
- Only authenticated users can access protected routes.
- Only the owner can mutate profile resources.

## Invariants

1. Idempotent Queue Operations: The database layer enforces unique idempotency_key string indices on incoming task check-offs to guarantee that network sync retries from offline clients never cause duplicate actions or logs.
2. No Background Tasks in Requests: Next.js route request handlers must completely hand off long-running operations (like AI image processing) to asynchronous background workers or secondary services to keep server response latency snappy.
3. Ephemeral Public Data Access: Shared vet report links generated by users function under strict short-circuit validation checks; they bypass standard account login gates but self-destruct from the cache storage array exactly 2 hours post-creation.
4. Metadata and large generated artifacts are stored in separate layers.
5. Auth and ownership are enforced at every mutation boundary.
6. Client components are used only where browser interactivity or real-time state requires them.
